Transporti V1 · Frontend Next.js

Audit UX/UI complet

Revue menée le 12 juillet 2026 · 4 axes analysés en parallèle : design system, parcours utilisateur, i18n/RTL, accessibilité & responsive · ~120 fichiers frontend

Les fondations sont saines — palette de marque définie, police Cairo pour l'arabe, parité quasi parfaite des traductions fr/ar, patterns modernes (skeletons, cartes lazy, bottom nav mobile). Mais l'absence de composants de base (Button, Input, Modal) et la généralisation incomplète des bons patterns produisent une dérive systémique : chaque page réinvente ses styles, ses états d'erreur et ses dialogues.

Verdict global : l'app est fonctionnelle en français/LTR sur les parcours principaux, mais pas prête pour un public arabophone (RTL fragile, 2/3 des écrans non traduits) ni pour une mise en production du flux de paiement en l'état (champs carte jamais transmis, faux ID de transaction, promesse de redirection non tenue).

Le point fort à protéger : le parcours post-livraison (stepper de mission, confirmation escrow, avis) est le mieux conçu de l'app — c'est le standard de qualité à généraliser vers l'amont.

1 · Design system

Ce qui existe : palette de marque (bleu royal brand #1E3A8A, vert accent, orange cta), tokens d'espacement base 8 px, z-index sémantiques, Inter + Cairo via next/font, dark mode… mais uniquement sur l'admin.

  • Aucune primitive UI. Pas de Button, Input, Card, Badge génériques — components/ui/ ne contient que 8 utilitaires. Chaque page recopie de longues chaînes Tailwind, d'où toutes les dérives ci-dessous.
  • Le CTA primaire est-il vert ou bleu ? Auth = vert accent-500/600, app = bleu brand-600, dashboard = vert, landing = orange cta-500. Décision de marque à trancher, puis à encoder dans un composant Button.
  • 3 systèmes de badges de statut contradictoires (ui/StatusBadge, admin/StatusBadge, lib/format.ts:getStatusColor) : le même statut « PUBLISHED » est vert, orange ou cyan selon la page. Labels FR en dur dans les trois.
  • Doublons : 2 Pagination, 2 StatCard (dont un hex hardcodé #1e293b), 2 systèmes de skeleton, keyframes dupliquées entre tailwind.config.ts et globals.css.
  • 3 échelles de bleu concurrentes (brand, primary, blue-* natif — 49 usages bruts dans 26 fichiers) et 84 hex littéraux dans les composants.
  • Dark mode déconnecté : 0 dark: dans app/(app)/ ; les variables CSS --color-neutral-* inversées en .dark sont inopérantes car Tailwind est configuré en hex statiques.

2 · Parcours utilisateur

  • Auth : mot de passe min. 6 à l'inscription vs min. 8 au reset ; username dérivé de l'email (collisions) ; pas de validation du téléphone tunisien ; boutons sociaux absents de /register ; logout sans redirection ; reset-password non traduit.
  • Création de job : règle de date contradictoire (attribut min = +24 h, JS = +15 min, texte d'aide = 24 h, calcul en UTC → décalage d'1 h) ; champ postal_code fantôme non initialisé dans le state ; pas de carte interactive pour pointer la destination (le GPS ne sert que pour la position actuelle) ; pas de lien « Modifier » par section sur l'écran de confirmation.
  • Offres : le client ne peut pas décliner une offre (accepter ou ignorer seulement) — le transporteur reste « En attente » jusqu'à expiration.
  • Photos : erreurs d'upload écrasées les unes par les autres — impossible de savoir quelles photos ont été rejetées.
  • Point fort : le flux post-livraison (stepper → confirmation client → libération escrow → avis, avec gate de vérification transporteur) est le standard de qualité de l'app.

3 · i18n & RTL (arabe)

  • Mécanisme sain : provider maison typé, ar.ts code-splitté, persistance localStorage, dir/lang posés dynamiquement. Parité excellente : les 848 clés fr existent toutes en ar (une section morte offersTabs à supprimer dans ar.ts:450).
  • Mais couverture réelle ~1/3 : 39 fichiers sur ~119 utilisent le hook. Estimation 400–600 chaînes en dur, dont 33 placeholders, 7 aria-labels (mélange FR/EN), 17 toasts et les confirm() natifs.
  • RTL : zéro propriété logique Tailwind (ms-/me-/ps-/pe-) dans le code ; le patch global globals.css:539-558 contient des règles mortes (un sélecteur qui applique du padding au lieu de margin, un no-op flex-row) ; la classe rotate-flip définie pour les icônes est utilisée 0 fois.
  • SSR : useAppI18n.tsx:109 fait return null tant que la locale n'est pas chargée → flash blanc, rien côté SSR, et <html lang="fr"> statique → FOUC de direction pour les arabophones.
  • Formats : ~25 dates en fr-FR/fr-TN hardcodées même en arabe ; devise jamais via Intl.NumberFormat.

4 · Accessibilité & responsive

  • Critique : labels non associés hors auth (paiement inclus) ; modales sans sémantique ni clavier (sauf ConfirmModal) ; StarRating de l'avis inutilisable au lecteur d'écran (5 boutons sans nom, focus supprimé) ; boutons-icônes muets (supprimer photo, afficher mot de passe, actions admin).
  • Majeur : contrastes sous WCAG AA (text-neutral-400 ≈ 2.5:1 pour du texte porteur d'information, mentions escrow incluses) ; erreurs jamais annoncées (role="alert" absent) ; pas de style global :focus-visible ni de skip-link ; hamburger sans aria-expanded.
  • Mobile : cibles tactiles < 44 px (checkboxes admin ~20 px, chips de suggestion ~28 px) ; tables admin 9 colonnes en scroll horizontal avec les actions hors écran ; modale de réservation sans max-h/overflow-y-auto (footer coupé avec le clavier virtuel) ; sidebar de filtres qui pousse les résultats sous la ligne de flottaison.
  • Performance perçue : bonne base (next/image majoritaire, cartes en dynamic import, 19 fichiers avec skeletons) ; à corriger : <img> natif dans PhotoUploader et DocumentReviewDrawer, fetch browse sans AbortController (réponses désordonnées sur 3G), « loading » texte brut sur browse.

Feuille de route suggérée

Sprint 1 · Confiance & bugs réels

Réparer ce qui trompe l'utilisateur

Flux de paiement (brancher réellement ou simplifier honnêtement) · faux états vides → bannière erreur + réessayer · classes CSS mortes (error-*, warning-*, classes dynamiques LocationPicker) · validation par étape du wizard + brouillon localStorage · bouton supprimer photo visible et ≥ 44 px.

Sprint 2 · Fondations design system

Créer les primitives et trancher la marque

Décider la couleur du CTA primaire (vert ou bleu) · créer Button, Input, Card, Modal accessible (généraliser le pattern ConfirmModal) · unifier les 3 StatusBadge sur i18n · fusionner Pagination/StatCard/Skeleton · remplacer confirm()/prompt() par les modales.

Sprint 3 · Marché arabophone

i18n complet + RTL propre

Internationaliser les pages critiques (jobs/[id], jobs/new, messages, booking, verification) · migrer vers les propriétés logiques Tailwind et supprimer le patch CSS · helpers formatDate/formatTND avec Intl · flipper les icônes directionnelles · SSR de la locale (cookie) pour éliminer le FOUC.

Sprint 4 · Finitions a11y & mobile

Généraliser les bons patterns

Labels htmlFor/id partout · :focus-visible global + suppression des outline-none orphelins · contrastes (neutral-400 → 500/600) · role="alert" sur les erreurs · vue mobile des tables admin · filtres browse complets (24 gouvernorats + destination + URL) et repliables sur mobile.
